# Guion de Explicación de Páginas - HomeCredit

## 1. Página de Clientes

La página de Clientes presenta una interfaz visual organizada con un encabezado que incluye el título "Clientes" y un botón destacado "Nuevo Cliente" con icono de usuario. Visualmente, cada cliente se muestra en tarjetas individuales con un diseño limpio que incluye un avatar circular con colores únicos asignados automáticamente (o una imagen del distrito si está disponible), seguido de la información principal del cliente: nombre completo, tipo y número de documento, email, teléfono, ingreso mensual y estado civil. Cada tarjeta tiene botones de edición y eliminación en la esquina superior derecha. A nivel de base de datos, los clientes se almacenan en la tabla `clients` de Firebase/Supabase, que contiene campos como `id`, `user_id` (vinculado al usuario autenticado), `document_type`, `document_number`, `full_name`, `email`, `phone`, `marital_status`, `dependents`, `monthly_income`, `color` (índice para asignación de colores), y campos de ubicación (`district`, `province`, `department`). La consulta filtra los clientes por `user_id` para mostrar solo los del usuario actual, y se ordenan por fecha de creación descendente. Al crear un nuevo cliente, el sistema asigna automáticamente un color basado en el número total de clientes existentes, y al eliminar, se realiza una eliminación directa en la base de datos.

## 2. Página de Propiedades

La página de Propiedades muestra un diseño en cuadrícula (grid) con tarjetas visuales atractivas que incluyen una imagen superior del distrito (si está disponible) o un icono por defecto, seguida de información detallada: nombre del proyecto, número de unidad, dirección completa, área en metros cuadrados, precio con su moneda correspondiente (Soles o Dólares), tipo de propiedad (Departamento, Casa o Dúplex) y un badge de estado (Disponible, Reservado, Vendido, o "No disponible" si está asignada a una simulación). Cada tarjeta tiene botones de edición y eliminación superpuestos en la esquina superior derecha. A nivel de base de datos, las propiedades se almacenan en la tabla `property_units` con campos como `id`, `user_id`, `property_name`, `unit_number`, `address`, `district`, `province`, `department`, `property_type`, `total_area`, `price`, `currency`, y `status`. La página realiza una consulta combinada que obtiene tanto las propiedades como las simulaciones existentes para determinar qué propiedades están asignadas (tienen `credit_simulations` asociadas), mostrando el estado "No disponible" visualmente. Al eliminar una propiedad que tiene simulaciones asociadas, el sistema primero elimina los `payment_schedules` relacionados, luego las `credit_simulations`, y finalmente la propiedad, manteniendo la integridad referencial de los datos.

## 3. Página de Documentación

La página de Documentación presenta una interfaz de acordeón interactiva con secciones expandibles que incluyen iconos distintivos y títulos descriptivos. Visualmente, cada sección tiene un encabezado con un icono en un fondo azul claro, el título de la sección, y un chevron que indica si está expandida o colapsada. Al expandir una sección, se muestra contenido estructurado con listas, párrafos informativos, alertas con colores (azul para información, verde para buenas prácticas, amarillo para tips, púrpura para consejos profesionales), e iconos que acompañan cada elemento. Las secciones principales incluyen: Introducción a HomeCredit, Dashboard, Gestión de Clientes, Administración de Propiedades, Simulaciones de Crédito, y Preguntas Frecuentes. A nivel de base de datos, esta página es principalmente estática y no realiza consultas directas a la base de datos, ya que su contenido es informativo y estático definido en el código del componente. Sin embargo, conceptualmente, la documentación explica cómo interactuar con las tablas `clients`, `property_units`, `credit_simulations`, y `payment_schedules`, proporcionando guías sobre cómo los usuarios deben gestionar sus datos almacenados en estas estructuras.

## 4. Landing Page

La Landing Page es una página de presentación visualmente impactante con un diseño de una sola página (one-page) que incluye múltiples secciones: un hero section con un diseño de dos columnas que presenta el logo de HomeCredit, un título grande "Gestión Inteligente de Créditos Hipotecarios", descripción del servicio, características destacadas con checkmarks verdes, y un botón centralizado de acceso con icono; una sección del equipo con tarjetas de miembros mostrando fotos (o emojis por defecto), nombres, roles y códigos universitarios; una sección explicativa del proceso de simulación en 4 pasos con iconos y descripciones; una sección de planes de precios con tarjetas diferenciadas por colores (azul, verde, púrpura, naranja) mostrando características y precios; y un footer completo con enlaces, logo y texto grande "HomeCredit". A nivel de base de datos, esta página no realiza consultas directas ya que es una página pública de presentación, pero conceptualmente prepara a los usuarios para interactuar con el sistema que utiliza Firebase Authentication (tabla `auth.users`) y las tablas de datos (`profiles`, `clients`, `property_units`, `credit_simulations`) una vez que se registren o inicien sesión. El botón principal redirige a la página de autenticación, que es el punto de entrada al sistema con base de datos.

## 5. Página de Login

La página de Login presenta un diseño de dos columnas: la izquierda (visible en pantallas medianas y grandes) muestra información promocional de HomeCredit con el logo, título, descripción del sistema y una lista de características con puntos azules; la columna derecha contiene el formulario de inicio de sesión en una tarjeta blanca con sombra, que incluye un icono de login en un círculo azul, el título "Iniciar Sesión", campos de email y contraseña con etiquetas, un botón de envío azul, y un enlace para cambiar al formulario de registro. Visualmente, la página tiene un fondo degradado de azul a verde, y el formulario muestra mensajes de error en rojo si las credenciales son incorrectas. A nivel de base de datos, el login utiliza Firebase Authentication para autenticar al usuario mediante `supabase.auth.signInWithPassword()`, que consulta la tabla `auth.users` de Firebase. Una vez autenticado exitosamente, el sistema carga el perfil del usuario desde la tabla `profiles` usando el `user_id`, lo que permite personalizar la experiencia y determinar el rol del usuario (admin, advisor, o client). Si las credenciales son incorrectas, se muestra un mensaje de error traducido al español, y el usuario permanece en la página sin acceso a los datos.

## 6. Página de Registro

La página de Registro comparte el mismo diseño de dos columnas que el Login, pero el formulario de registro está en la columna derecha con un diseño similar pero con colores verdes en lugar de azules (icono verde, botón verde). El formulario incluye campos para nombre completo, correo electrónico, contraseña (con validación de mínimo 6 caracteres mostrada en tiempo real), confirmación de contraseña, un botón de registro verde, y un enlace para cambiar al formulario de login. Visualmente, se muestran validaciones en tiempo real (como advertencia si la contraseña tiene menos de 6 caracteres) y mensajes de error específicos si el email ya está registrado o si hay problemas de validación. A nivel de base de datos, el registro realiza dos operaciones principales: primero, crea un nuevo usuario en Firebase Authentication mediante `supabase.auth.signUp()`, que inserta un registro en `auth.users` con el email y una contraseña hasheada; segundo, inmediatamente después de la creación exitosa del usuario, inserta un registro en la tabla `profiles` con el `id` del usuario (que coincide con el `user_id`), `email`, `full_name`, y `role` (por defecto 'advisor'). Esta doble inserción asegura que cada usuario tenga tanto credenciales de autenticación como un perfil completo en el sistema, permitiendo que el usuario pueda acceder inmediatamente a todas las funcionalidades de gestión de clientes, propiedades y simulaciones.
